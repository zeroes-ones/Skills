# Payment Infrastructure Security

ISO 8583 message security, HSM-based PIN protection and EMV chip
authentication patterns for card-based payment networks.

## ISO 8583 Message Security

ISO 8583 is the financial transaction card-originated messages standard
used between acquirer, issuer, and network (Mastercard/Visa/Amex).

### Message Structure
Each message has: MTI (Message Type Indicator) + bitmap + data elements.
Key security-relevant data elements:

| DE | Name | Sensitivity | Handling |
|----|------|-------------|----------|
| DE 2 | PAN | Confidential | Encrypt in transit, mask in logs (first 6 + last 4) |
| DE 14 | Expiry Date | Low (alone) | Mask in logs; combine with PAN = sensitive |
| DE 35 | Track 2 Data | CRITICAL | NEVER store post-authorization; P2PE end-to-end |
| DE 45 | Track 1 Data | CRITICAL | NEVER store post-authorization |
| DE 52 | PIN Block | CRITICAL | HSM-encrypted, never in clear, format ISO 9564 |
| DE 55 | EMV Data (ICCD) | High | Contains cryptogram, unpredictable number; protect integrity |

### End-to-End Perspectives

**Acquirer Side (Terminal → Acquirer → Network):**
```
Terminal reads chip/magstripe → DE 52 (PIN block encrypted with DUKPT)
  → Acquirer decrypts DUKPT → re-encrypts with ZPK (Zone PIN Key) under LMK
  → Forward to network (Mastercard/Visa) → network translates from ZPK to issuer's ZPK
  → Issuer decrypts with LMK → validates PIN in HSM
```

**Key Management (DUKPT — Derived Unique Key Per Transaction):**
- BDK (Base Derivation Key): provisioned into terminal at manufacture, never leaves HSM
- KSN (Key Serial Number): sent with each transaction, identifies derivation path
- IPEK (Initial PIN Encryption Key): derived from BDK + KSN
- Per-transaction key: derived from IPEK, unique per transaction, forward secrecy

## HSM Architecture

Hardware Security Modules for PIN processing and key management.

### PIN Translation (ISO 9564)
```
PAN → format to 12 digits (rightmost excluding Luhn check digit)
   ↓
PAN + PIN → XOR → pad with 'F's → encrypt under working key (PIN block)
   ↓
Translation at interchange: decrypt under ZPK-1 → re-encrypt under ZPK-2
                            (inside HSM only — never in clear)
```

### HSM Commands (Thales Payshield)
| Command | Purpose |
|---------|---------|
| A0 (Generate Key) | Generate random key component, combine with others for split knowledge |
| BA (Translate PIN) | Decrypt PIN block under one key, re-encrypt under another (in HSM) |
| CA (Verify PIN) | Decrypt PIN block, compare with customer PIN (thwart PIN generation attack) |
| DA (Generate PIN) | Generate random PIN for new card issuance |
| EA (Verify ARQC) | Validate EMV Application Request Cryptogram |
| FA (Generate ARPC) | Generate EMV Application Response Cryptogram (issuer response) |

### Key Types (Payshield)
```
001 = ZMK (Zone Master Key) — encrypts key exchange
002 = ZPK (Zone PIN Key) — encrypts PIN blocks between parties
008 = BDK (Base Derivation Key) — DUKPT base key
00A = MAC Key — generates/verifies message authentication codes
00B = DEK (Data Encryption Key) — encrypts sensitive data fields
```

## EMV Security

### Offline Data Authentication (ODA)

| Method | What It Verifies | How |
|--------|-----------------|-----|
| **SDA** (Static) | Card data not modified since issuance | Terminal validates CA + issuer certificate chain; card provides static signed data |
| **DDA** (Dynamic) | Card is not cloned + data not modified | Card generates dynamic signature using ICC private key; terminal verifies with ICC public key + challenge |
| **CDA** (Combined) | DDA + transaction integrity | Dynamic signature covers transaction details (amount, currency, unpredictable number) — prevents relay attacks |

### Cryptogram Types
```
ARQC (Authorization Request Cryptogram): Card → Issuer
  Generated by card from: transaction data + unpredictable number + ATC
  Issuer verifies ARQC → returns ARPC if approved
  If ARQC fails validation → decline (possible counterfeit card)

ARPC (Authorization Response Cryptogram): Issuer → Card
  Generated by issuer HSM, verified by card chip
  If ARPC fails → card refuses to complete transaction

AAC (Application Authentication Cryptogram): Card → Terminal (decline)
  Card declines offline (e.g., floor limit exceeded, risk management)
  No online request sent — terminal declines immediately
```

### Contactless Security (EMVCo)
```
Relay Attack Mitigation:
  - Transaction limits: contactless without CVM typically €50/£100
  - Cumulative limits: require chip contact + PIN after cumulative threshold
  - Proximity check: timing constraints (response must arrive within ~2ms)
  - Terminal velocity: same terminal, rapid succession → force chip contact

Mag-stripe Fallback Attack (requires chip cards to also have magstripe):
  - Terminal should prefer chip; if chip fails, require magstripe CVV/CVC check
  - Track 2 Service Code position 2: value 2 or 6 = chip card
    If Service Code says chip but magstripe read → flag as potential counterfeit
```
