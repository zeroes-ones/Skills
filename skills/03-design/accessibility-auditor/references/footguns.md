# Footguns

<!-- DEEP: 10+min — war stories from production accessibility audits -->

| Footgun | What Happened | Root Cause | How to Prevent |
|---------|---------------|------------|----------------|
| Automated a11y scan scored 98/100 — but manual keyboard testing found 340 critical blockers because axe-core only catches ~30% of WCAG issues | A fintech company celebrated a "passing" audit after integrating axe-core into CI and achieving a Lighthouse score of 98. Their VP of Product tweeted about being "WCAG 2.2 AA compliant." Three weeks later, a blind customer's lawsuit revealed that 47 modal dialogs didn't trap focus, 12 custom dropdowns were invisible to VoiceOver, and the checkout flow required mouse hover to reveal the "Place Order" button. Settlement: $85,000. | Automated tools are syntax checkers, not accessibility validators. axe-core catches missing alt text, color contrast, and duplicate IDs — it cannot detect whether `role="combobox"` actually behaves like a combobox, whether focus lands in a sensible order, or whether error messages are programmatically associated with form fields during real interaction. ~70% of WCAG success criteria require human judgment. | **Treat automated scans as a floor, not a ceiling.** The 30% rule: automated tools catch ~30% of accessibility issues. Allocate testing time proportionally: 20% automated, 40% manual keyboard, 40% screen reader. Require manual testing sign-off on every PR that changes interactive components. Run axe-core as a gate ("no new violations") but never report "we're accessible" based on automation alone. The dashboard should say "No automated violations detected — manual review required." |
| `aria-label="Close dialog"` overrode visible text "Submit Application" in 14 languages — screen reader users heard contradictory instructions and abandoned forms | A SaaS platform served 14 locales. The French translation of "Submit Application" was "Soumettre la candidature." The ARIA label `aria-label="Close dialog"` was hardcoded in English in the component library. French screen reader users heard "Close dialog — Soumettre la candidature" and abandoned the form, thinking it would discard their data. Support received 230 tickets from non-English users over 6 weeks before anyone connected the issue to the aria-label. | ARIA labels were defined as string literals in component source code, completely separate from the i18n pipeline. The accessibility team and localization team never spoke — aria-labels weren't in the translation management system. Visible text and accessible names diverged silently across every language except English. | **Treat `aria-label` as user-facing text that must go through the i18n pipeline.** Use `aria-labelledby` instead of `aria-label` whenever visible text exists — it references the existing translated element by ID, so translations stay in sync automatically. Audit rule: "Does every `aria-label` have a corresponding string in the translation files?" Add a lint rule that blocks `aria-label` on elements that already have visible text content. |
| Focus trap worked perfectly on Chrome/Mac but on iOS Safari with VoiceOver, focus escaped the modal and landed on a disabled "Delete Account" button in the page background | An e-commerce checkout modal trapped focus using a standard `focus-trap` library. QA tested on desktop — Chrome, Firefox, Edge — and signed off. But iOS Safari's VoiceOver uses a separate accessibility focus cursor that doesn't respect DOM focus traps in the same way. Blind users on iPhones swiped past the modal boundary and VoiceOver announced "Delete Account — button — dimmed" in the middle of checkout. Mobile conversion dropped 4.2% on iOS over 3 months before anyone tested with a real iPhone and VoiceOver. | The testing matrix covered browsers and viewports but not browser × assistive technology combinations. Focus trapping relies on `document.activeElement` tracking, but VoiceOver maintains its own accessibility tree that can navigate to elements with `aria-hidden="false"` even when they're visually obscured behind a modal overlay. | **Test focus traps on the platform × AT matrix, not just browser × viewport.** Required combinations: VoiceOver+Safari (iOS), VoiceOver+Safari (macOS), NVDA+Firefox (Windows), JAWS+Chrome (Windows), TalkBack+Chrome (Android). Use the `inert` attribute on background content (not just `aria-hidden`) to remove it from both DOM and accessibility tree simultaneously. Test by navigating with swipe/gesture, not just Tab key — screen reader users don't Tab. |
| `prefers-reduced-motion: reduce` was respected perfectly — but `prefers-reduced-transparency: reduce` was ignored, and 3 parallax sections with frosted glass effects triggered migraines for users with vestibular disorders | A design agency portfolio site followed WCAG 2.2 to the letter: all animations stopped when `prefers-reduced-motion: reduce` was set. But the site used backdrop-filter blur on 3 hero sections with parallax scrolling. These frosted-glass effects cause a "swimming" visual distortion when scrolling that triggers vestibular migraines. Users who had set BOTH `prefers-reduced-motion` and `prefers-reduced-transparency` still suffered because the latter preference was never checked. Three users filed ADA complaints citing the site as "physically painful to use." | WCAG 2.2 SC 2.3.1 (Three Flashes or Below Threshold) and 2.3.3 (Animation from Interactions) cover motion. Transparency/blur effects that cause visual distortion are not explicitly covered by a numbered SC — but they fall under the general principle of "content must not cause physical harm." The team checked the SC checklist but not the spirit of the standard. | **Respect all 5 `prefers-*` media queries: `prefers-reduced-motion`, `prefers-reduced-transparency`, `prefers-contrast`, `prefers-color-scheme`, `forced-colors`.** For vestibular safety: any backdrop-filter blur, parallax effect, or scroll-linked animation must disable when `prefers-reduced-motion` OR `prefers-reduced-transparency` is active. Test by enabling both OS-level settings and scrolling through every page — if you feel any discomfort, it fails. Document this testing step in the accessibility QA checklist. |
| Color contrast ratio passed WCAG AA on the design tool's flat swatches — but the actual production background was a gradient, and white text over the middle of the gradient failed 47 of 92 color stops | A healthcare dashboard used a blue-to-purple gradient as a hero background with white text. The Figma design showed white (#FFFFFF) on mid-blue (#3B82F6) — a 4.62:1 ratio, passing AA. In production, the CSS gradient had 92 color stops (smoothed by the browser). White text over stops 31-67 fell below 3:1 because those mid-gradient colors were significantly lighter. A patient with low vision couldn't read the "Book Appointment" CTA. The company paid a $15,000 settlement after a demand letter. | Contrast was checked on a single representative color from the gradient, not sampled across the full gradient range. Design tools (Figma, Sketch) report contrast against a fill layer — not against the actual rendered pixel output. CSS gradients are continuous color spaces; any point along the gradient is a potential contrast failure. | **Sample contrast at every 10% interval across any gradient background.** Write a script that extracts gradient CSS, samples colors at 0%, 10%, 20%, ..., 100% positions, and computes contrast ratio against the text color at each point. All samples must pass 4.5:1 (AA) or 7:1 (AAA) depending on your target. If the gradient is too light in the middle, add a subtle text-shadow or a semi-transparent overlay to maintain contrast. Never trust a single-point contrast check on a gradient background. |
