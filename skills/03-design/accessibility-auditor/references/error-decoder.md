| `grep -iP '(focus.order.*broken\|tab.*skip\|Tab.*doesn.*work\|cannot.*tab.to\|keyboard.*stuck\|focus.*lost)' logs/a11y-issues.log` | Keyboard navigation breaks: users can't Tab to critical controls, Tab order jumps unpredictably, or focus gets stuck in a component | Tab order follows DOM order unless overridden. `tabindex="0"` on wrapper divs, hidden elements still in tab order, or `tabindex="-1"` misused — all break the expected sequence | Audit tab order with `document.querySelectorAll('[tabindex]:not([tabindex="-1"]), a, button, input, select, textarea')`. Fix: remove positive tabindex values, use DOM source order, set inert/hidden elements to `tabindex="-1"` | 1. Open browser DevTools → Elements panel. 2. Run `document.body.focus()`. 3. Tab through the page, counting each focus ring. 4. For each focusable element: verify it receives focus AND is visible on screen. 5. If tabbing jumps: check for `tabindex="0"` on parent divs — remove them. 6. If element not in tab order: check `tabindex="-1"` and `display:none`/`visibility:hidden` parents. 7. Re-test after each fix |
| `grep -iP '(screen.reader.*silence\|NVDA.*nothing\|VoiceOver.*no.*announce\|JAWS.*unlabel\|no.accessible.name\|button.*without.*label)' logs/a11y-issues.log` | Screen reader announces "button" or "clickable" without a name — user can't determine what action will happen | Interactive element has no accessible name. `<button>` without text content, `<a>` without text or `aria-label`, `<input>` without associated `<label>`. WCAG 4.1.2 Name, Role, Value (Level A) | Provide accessible name via: text content (preferred), `<label>` with `for`, `aria-label`, or `aria-labelledby`. Text content is the most robust — it's always visible and always announced | 1. Open Chrome DevTools → Accessibility panel (in Elements tab). 2. Select the element. 3. Inspect "Computed Properties → Name". 4. If empty: add text content inside button/link, or associate `<label>`. 5. Test with VoiceOver (Cmd+F5 on Mac): navigate by button and verify it announces "[button name], button". 6. Test with NVDA (insert+F7 Elements List): verify button appears with a name. 7. NEVER add `aria-label` without testing — typos in aria-label are invisible and only caught by SR testing |
| `grep -iP '(color.contrast.*fail\|contrast.*too.*low\|grey.*on.*white\|light.*grey.*text\|cannot.*read.*color)' logs/a11y-issues.log` | Users can't read text: color contrast too low (e.g., light gray text on white background) | Text color contrast ratio < 4.5:1 (normal text) or < 3:1 (large text 18px+/bold 14px+). WCAG 1.4.3 Contrast Minimum (Level AA). Most common in design systems where "faint" text is used for secondary information | Use a contrast checking tool: `npx axe-core --rule color-contrast`. Text must have MINIMUM 4.5:1 ratio against background. For large text (≥18pt or bold ≥14pt): minimum 3:1 | 1. Open Chrome DevTools → inspect the text element. 2. Click the color swatch in Styles panel. 3. Chrome shows contrast ratio next to color picker. 4. If < 4.5:1: darken text or lighten background until ratio ≥ 4.5:1. 5. For secondary text: use a darker gray (e.g., #595959 on white = 7:1) instead of light gray (#999 = 2.85:1). 6. Update design tokens for all semantic text colors. 7. Add axe-core `color-contrast` to CI — fails build if any contrast violation found |
| `grep -iP '(missing.*heading\|heading.*skip\|h1.*missing\|multiple.*h1\|heading.*level.*skip)' logs/a11y-issues.log` | Screen reader user can't navigate by headings — heading levels skip (h1→h3 missing h2) or no headings exist | Headings used for visual styling (h4 looks smaller), not semantic structure. Developers choose heading level based on font size, not document hierarchy. WCAG 1.3.1 Info and Relationships (Level A) | Audit with `document.querySelectorAll('h1, h2, h3, h4, h5, h6')`. Verify heading hierarchy: exactly one `<h1>`, no level gaps (no h1→h3 without h2 between). Fix: style headings with CSS classes, not by choosing a deeper heading level | 1. Install HeadingsMap browser extension (Chrome/Firefox). 2. Navigate to page. 3. HeadingsMap shows full hierarchy — red highlights indicate gaps. 4. Each heading must have only 1 level deeper than its parent (h1→h2→h3). 5. If gap: change heading level to match hierarchy, use CSS class for visual styling. 6. If multiple h1s: make one `<h1>` for page title, rest become `<h2>`. 7. Re-run HeadingsMap after fix — tree must be green |
| `grep -iP '(modal.*not.*close\|Escape.*not.*work\|dialog.*stuck\|cannot.*close\|focus.not.return\|focus.*lost.on.close)' logs/a11y-issues.log` | Modal opens but can't be closed with keyboard — Escape doesn't work, focus lost on close, or focus returns to document top instead of trigger | Missing keyboard event handler: `onKeyDown` for Escape not implemented, `aria-modal="true"` missing, focus management logic incomplete. WCAG 2.4.3 Focus Order (Level A) | Implement complete modal lifecycle: (1) capture ref to trigger element, (2) on open: focus first focusable in modal, (3) on Escape/Space click outside: close modal and `triggerRef.focus()`, (4) `aria-modal="true"` traps screen reader focus | 1. Open modal with Enter/Space on trigger. 2. Press Tab repeatedly — focus must stay inside modal. 3. If focus escapes: add `aria-modal="true"` and a focus-trap library or custom `tabIndex` loop. 4. Press Escape — modal closes. 5. If not: add `onKeyDown(e) { if (e.key === 'Escape') handleClose(); }`. 6. After close: verify focus returns to trigger element — if not, fix `triggerRef.current?.focus()`. 7. Test full cycle 3x: open→tab→escape→close→refocus |
| `grep -iP '(form.*cannot.*submit\|error.*not.*announced\|validation.*silent\|error.*inaccessible\|required.*field.*missing)' logs/a11y-issues.log` | Form errors are invisible to screen reader users — errors displayed as red text only, no announcement, no aria-invalid | Error state only communicated visually (red border, red text, icon). No `aria-invalid`, no `aria-describedby` linking error to input, no programmatic focus management. WCAG 3.3.1 Error Identification (Level A) | Every error input: `aria-invalid="true"` + `aria-describedby="[error-id]"` pointing to a visible error message. On submit failure: move focus to first error field AND announce errors count via a live region | 1. Set `aria-invalid="true"` on inputs with errors. 2. Create error message element: `<span id="email-error" role="alert">Invalid email format</span>`. 3. Add `aria-describedby="email-error"` on the input. 4. On form submit: if errors exist, prevent submit, set `document.querySelector('[aria-invalid]').focus()`. 5. Add hidden live region: `<div aria-live="polite" aria-atomic="true">[N] form errors found</div>`. 6. Test with VoiceOver: submit invalid form — must announce "N form errors found" and focus moves to first error |
