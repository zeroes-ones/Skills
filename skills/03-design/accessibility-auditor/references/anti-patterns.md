# Anti-Patterns

<!-- QUICK: 90s -- 4-column machine-checkable format. Every anti-pattern has a grep to find it and a lint/prevention config. -->

| ❌ Anti-Pattern | ✅ Do This Instead | 🔍 Detect (grep / lint) | 🛡️ Auto-Prevent |
|-----------------|---------------------|--------------------------|-------------------|
| Running only automated accessibility tests (Lighthouse, axe-core) and declaring the product "accessible" | Add manual testing layers: keyboard-only navigation on all critical flows, screen reader testing with VoiceOver + NVDA, and testing with real users with disabilities | `grep -L 'manual.test\|keyboard.only\|screen.reader\|VoiceOver\|NVDA\|human.judgment' audit-report*.md` — reports missing manual testing evidence | Audit template: Add `## Manual Testing Results (Required)` section with keyboard flow, screen reader flow, and user testing subsections. Template validator blocks reports without manual results |
| Using `<div onclick="...">` instead of `<button>` because "styling buttons is hard" | Use native `<button>` and style it. `<button>` is focusable, keyboard-operable, and announces its role to screen readers for free | `grep -rn '<div[^>]*\bonclick=' *.html *.tsx *.jsx` — divs with click handlers | ESLint rule: `jsx-a11y/no-static-element-interactions: error` + `jsx-a11y/click-events-have-key-events: error`. CI fails on non-semantic click handlers |
| Adding `role="button"` to a `<div>` without keyboard handlers, focus styles, or disabled state | Implement the full keyboard interaction model: Enter/Space to activate, Tab to reach, visible focus indicator, `aria-disabled` for disabled state | `grep -rn 'role="button"' *.html *.tsx *.jsx \| grep -v 'onKeyDown\|onKeyUp\|tabIndex\|aria-disabled\|:focus'` — role=button without keyboard handlers | ESLint rule: `jsx-a11y/interactive-supports-focus: error` + `jsx-a11y/no-noninteractive-element-interactions: error`. Custom widget checklist: `scripts/check-aria-widget.sh` |
| Putting `aria-label` on `<div>` or `<span>` — ARIA labels are ignored on non-semantic elements unless role is also set | Put `aria-label` only on interactive elements (`<button>`, `<a>`, `<input>`) or elements with a widget `role`. For `<div>` sections, use `role="region"` with `aria-labelledby` | `grep -rn 'aria-label=' *.html *.tsx *.jsx \| grep -P '<(div\|span)\b' \| grep -v 'role='` — aria-label on div/span without role | ESLint rule: `jsx-a11y/aria-props: error` validates allowed ARIA on elements. axe-core `aria-allowed-attr` rule catches this in CI |
| Shipping a modal without focus-trap testing — focus not trapped, Escape doesn't close, focus not restored | Every modal must pass: (1) focus moves to first focusable on open, (2) Tab cycling stays inside, (3) Escape closes, (4) focus returns to trigger, (5) SR announces content | `grep -rn 'role="dialog"\|role="alertdialog"' *.html *.tsx *.jsx \| grep -v 'aria-modal\|focus-trap\|useFocusTrap\|Escape\|onClose'` — dialogs without focus management | Modal checklist: `scripts/check-modal-a11y.sh` verifies: focus trap, Escape key, aria-modal, focus restore, and visible close button. CI rejects PRs adding dialogs without this check |
| Using `outline: none` on focusable elements without providing a custom visible focus indicator | Provide a custom focus indicator (2px+ thickness, 3:1 contrast ratio) meeting WCAG 2.4.7 Focus Visible (Level AA) | `grep -rn 'outline\s*:\s*none\|outline\s*:\s*0' *.css *.scss \| grep -v 'box-shadow\|outline-offset\|:focus-visible'` — outline: none without visible replacement | CSS lint: `stylelint-a11y` rule `@a11y/focusable-selectors` catches missing focus styles. CI quality gate: axe-core `focusable-selectors` check |
| Labeling form inputs with placeholder text only — no `<label>`, no `aria-label`, no `aria-labelledby` | Every form input must have an associated `<label>` with `for` matching input `id`. Placeholder is a hint, not a label | `grep -rn '<input\|<textarea' *.html *.tsx *.jsx \| grep 'placeholder=' \| grep -v 'id="\|\<label' \| grep -v 'aria-label='` — inputs with placeholder but no label | axe-core rule `label` (WCAG 3.3.2). CI pipeline: `npx axe-core --rules label,label-title-only` fails builds. ESLint: `jsx-a11y/label-has-associated-control: error` |
| Treating accessibility as a final QA gate — auditing after the feature is built | Shift left: audit designs for contrast, audit code during PR for semantics, audit builds in CI with automated tools | `grep -L 'accessibility\|a11y' .github/pull_request_template.md CODEOWNERS` — no a11y review step in PR process | CI quality gate: add `npx axe-core` check to CI pipeline. PR template: add `## Accessibility: [ ] Axe found 0 violations, [ ] Manual keyboard pass, [ ] SR pass`. CODEOWNERS: add `* @a11y-reviewer` for a11y-critical paths |
