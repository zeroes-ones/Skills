# Security Engineer - OWASP ASVS Checklist

OWASP Application Security Verification Standard (ASVS) Level 1 checklist for basic application security.

---

## How to Use This Checklist

- **Level 1 (Basic):** All applications should meet these requirements. Suitable for low-assurance applications.
- **Level 2 (Standard):** Applications handling sensitive data or business logic. Adds depth.
- **Level 3 (Advanced):** High-value applications (financial, healthcare, critical infra).

This document covers **Level 1** requirements. Mark each as Pass/Fail/Not Applicable.

## V1: Architecture, Design & Threat Modeling

| ID | Requirement | Status |
|----|------------|--------|
| 1.1.1 | Verify use of a secure software development lifecycle (SSDLC) | ☐ |
| 1.1.2 | Verify threat modeling for critical authentication, access control, business logic, and key flows | ☐ |
| 1.1.3 | Verify all user stories and features contain functional security constraints | ☐ |
| 1.1.4 | Verify documentation of all trust boundaries, components, and significant data flows | ☐ |
| 1.1.5 | Verify definition and enforcement of high-level architecture for the application and connected services | ☐ |
| 1.2.1 | Verify all components are up to date (no EOL software) and only required features enabled | ☐ |
| 1.2.2 | Verify all third-party components come from trusted, maintained repositories | ☐ |
| 1.2.3 | Verify a software bill of materials (SBOM) is maintained for all third-party libraries | ☐ |
| 1.2.4 | Verify the attack surface is documented and minimized | ☐ |

## V2: Authentication

### V2.1 Password Security
| ID | Requirement | Status |
|----|------------|--------|
| 2.1.1 | Passwords must be at least 8 characters | ☐ |
| 2.1.2 | Passwords up to 64+ characters allowed (no max truncation) | ☐ |
| 2.1.3 | All Unicode characters allowed in passwords (with normalization) | ☐ |
| 2.1.4 | Password strength meter provided to user | ☐ |
| 2.1.5 | No password hints or knowledge-based recovery | ☐ |
| 2.1.6 | Brute force protection: account lockout or progressive delays after 5-10 failed attempts | ☐ |
| 2.1.7 | Password change requires current password verification | ☐ |
| 2.1.8 | Passwords stored with adaptive salted hashing (bcrypt, argon2id, PBKDF2) | ☐ |
| 2.1.9 | No default passwords for any accounts | ☐ |
| 2.1.10 | Password reset tokens are random, single-use, expire within 20 minutes | ☐ |

### V2.2 General Authentication
| ID | Requirement | Status |
|----|------------|--------|
| 2.2.1 | Anti-automation controls on login, reset, MFA setup | ☐ |
| 2.2.2 | Generic error messages: no "user exists" vs "wrong password" distinction | ☐ |
| 2.2.3 | Credential recovery doesn't reveal current password | ☐ |
| 2.2.4 | MFA enabled for all user accounts (TOTP, FIDO2, WebAuthn) | ☐ |
| 2.2.5 | Re-authentication required for sensitive operations (change email, delete account, transfer) | ☐ |

## V3: Session Management

| ID | Requirement | Status |
|----|------------|--------|
| 3.1.1 | Session token never exposed in URL parameters | ☐ |
| 3.1.2 | Session token generated by cryptographically secure PRNG (min 128 bits entropy) | ☐ |
| 3.2.1 | Session invalidated on server after logout | ☐ |
| 3.2.2 | Idle timeout: sessions expire after 30 minutes inactivity | ☐ |
| 3.2.3 | Absolute timeout: sessions expire after 12 hours regardless of activity | ☐ |
| 3.3.1 | Token refreshed on login, role change, or privilege escalation | ☐ |
| 3.5.1 | Session cookies: Secure, HttpOnly, SameSite=Lax (Strict for sensitive) | ☐ |
| 3.5.2 | Session cookie: Path=/ for domain-wide; Domain attribute not set (restrict to origin) | ☐ |

## V4: Access Control

| ID | Requirement | Status |
|----|------------|--------|
| 4.1.1 | Access control enforced on trusted server-side layer (not client-only) | ☐ |
| 4.1.2 | Deny by default: all access denied unless explicitly granted | ☐ |
| 4.1.3 | Principle of least privilege applied to all users and services | ☐ |
| 4.1.4 | Ownership verified before read/update/delete (IDOR prevention) | ☐ |
| 4.1.5 | Directory listing disabled on web servers | ☐ |
| 4.2.1 | All API endpoints and HTTP methods have authorization checks | ☐ |
| 4.2.2 | JWT/oauth tokens validated for signature, expiry, audience, issuer | ☐ |
| 4.3.1 | Sensitive data access logged with actor, action, resource, timestamp | ☐ |
| 4.3.2 | Administrative actions require re-authentication or step-up auth | ☐ |

## V5: Validation, Sanitization & Encoding

### V5.1 Input Validation
| ID | Requirement | Status |
|----|------------|--------|
| 5.1.1 | All input validated server-side (not only client-side) | ☐ |
| 5.1.2 | Input validated against allowlist of acceptable values where possible | ☐ |
| 5.1.3 | Structured data (JSON, XML, CSV) validated against formal schema | ☐ |
| 5.1.4 | URL redirects validated against allowlist of destinations | ☐ |
| 5.1.5 | File uploads: validate type by magic bytes, limit size, scan for malware | ☐ |

### V5.2 Sanitization & Sandboxing
| ID | Requirement | Status |
|----|------------|--------|
| 5.2.1 | HTML sanitization used for user-supplied markup (DOMPurify, OWASP Java HTML Sanitizer) | ☐ |
| 5.2.2 | Unstructured data sanitized to remove unsafe characters | ☐ |
| 5.2.3 | User input sanitized before passing to mail systems (SMTP injection prevention) | ☐ |
| 5.2.4 | Avoid eval() and dynamic code execution with user input | ☐ |

### V5.3 Output Encoding & Injection Prevention
| ID | Requirement | Status |
|----|------------|--------|
| 5.3.1 | Output encoded for HTML context (HTML entity encoding) | ☐ |
| 5.3.2 | Output encoded for JavaScript context (Unicode escaping) | ☐ |
| 5.3.3 | Output encoded for CSS context | ☐ |
| 5.3.4 | SQL queries use parameterization (PreparedStatements, bind variables) | ☐ |
| 5.3.5 | OS commands use parameterization; avoid shell when using exec() | ☐ |
| 5.3.6 | LDAP queries escaped with OWASP or LDAP-specific encoding | ☐ |
| 5.3.7 | Content Security Policy (CSP) deployed to mitigate XSS impact | ☐ |
| 5.3.8 | Anti-CSRF tokens on all state-changing requests | ☐ |

## V6: Stored Cryptography

| ID | Requirement | Status |
|----|------------|--------|
| 6.1.1 | Sensitive data classified and encrypted at rest (database, file system, backups) | ☐ |
| 6.1.2 | Key management: keys stored separately from data (KMS, HSM, vault) | ☐ |
| 6.2.1 | All cryptographic modules use approved algorithms (AES-256-GCM, ChaCha20-Poly1305) | ☐ |
| 6.2.2 | No deprecated algorithms: MD5, SHA-1, RC4, DES, 3DES, ECB mode | ☐ |
| 6.2.3 | Cryptographic keys managed with secure random generation (min 128 bits) | ☐ |
| 6.2.4 | Encryption keys rotated according to policy (90-day max) | ☐ |
| 6.2.5 | Secrets never hardcoded in source code, config files, or CI/CD pipelines | ☐ |

## V7: Error Handling & Logging

### V7.1 Log Content
| ID | Requirement | Status |
|----|------------|--------|
| 7.1.1 | Logs written for: auth events, access control failures, input validation failures, errors | ☐ |
| 7.1.2 | Log entries include: timestamp, event type, source IP, user ID, resource, outcome | ☐ |
| 7.1.3 | No sensitive data in logs: passwords, tokens, PII, session IDs (hash instead) | ☐ |
| 7.1.4 | Log integrity protected: append-only storage, hash chaining, WORM | ☐ |
| 7.1.5 | Logs shipped to centralized, access-controlled log management system | ☐ |

### V7.2 Error Handling
| ID | Requirement | Status |
|----|------------|--------|
| 7.2.1 | Generic error messages displayed to users (no stack traces, SQL errors, paths) | ☐ |
| 7.2.2 | Unhandled exceptions caught by global error handler | ☐ |
| 7.2.3 | Debug mode disabled in production | ☐ |

### V8.1 Client-Side Data Protection

| ID | Requirement | Status |

| 8.2.2 | Data retained only as long as necessary (data retention policy) | ☐ |
| 8.2.3 | Users can request data deletion (right to erasure compliance) | ☐ |
| 8.2.4 | Sensitive data redacted/masked in non-production environments | ☐ |

## V9: Communications

### V9.1 Client-Server Communication
| ID | Requirement | Status |
|----|------------|--------|
| 9.1.1 | TLS 1.3 (minimum TLS 1.2) for all client-server connections | ☐ |
| 9.1.2 | Certificate signed by trusted CA; valid and not expired | ☐ |
| 9.1.3 | HTTP Strict Transport Security (HSTS) header with max-age ≥ 1 year, includeSubDomains | ☐ |
| 9.1.4 | All HTTP traffic redirected to HTTPS (301 permanent redirect) | ☐ |
| 9.1.5 | No mixed content (HTTPS page loading HTTP resources) | ☐ |

### V9.2 Server-Server Communication
| ID | Requirement | Status |
|----|------------|--------|
| 9.2.1 | Internal service communication encrypted (mTLS or service mesh) | ☐ |
| 9.2.2 | Valid TLS certificates for internal services (no self-signed in production) | ☐ |

## V10: Malicious Code Protections

| ID | Requirement | Status |
|----|------------|--------|
| 10.1.1 | Web Application Firewall (WAF) deployed at trust boundary | ☐ |
| 10.2.1 | Content Security Policy (CSP) header deployed with strict directives | ☐ |
| 10.2.2 | Third-party scripts from CDNs use Subresource Integrity (SRI) hashes | ☐ |
| 10.2.3 | X-Content-Type-Options: nosniff header set | ☐ |
| 10.2.4 | X-Frame-Options: DENY or SAMEORIGIN (or frame-ancestors CSP) | ☐ |

| 10.2.6 | Permissions-Policy header restricts browser features | ☐ |
| 10.3.1 | Dependencies audited for known vulnerabilities (automated in CI/CD) | ☐ |
| 10.3.2 | SBOM generated and reviewed per release | ☐ |

## Sample Security Headers Configuration

```nginx
# Recommended security headers
add_header Strict-Transport-Security "max-age=31536000; includeSubDomains; preload" always;
add_header Content-Security-Policy "default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline'; img-src 'self' data:; frame-ancestors 'none'; form-action 'self';" always;
add_header X-Content-Type-Options "nosniff" always;
add_header X-Frame-Options "DENY" always;
add_header Referrer-Policy "strict-origin-when-cross-origin" always;
add_header Permissions-Policy "camera=(), microphone=(), geolocation=()" always;
add_header Cache-Control "no-store" always;
```

## Quick Verification Commands

```bash
# Check SSL/TLS configuration
openssl s_client -connect example.com:443 -tls1_2
testssl.sh https://example.com

# Check security headers
curl -sI https://example.com | grep -iE 'strict-transport|content-security|x-frame|x-content|referrer-policy|permissions-policy'

# Check CSP for unsafe directives
curl -sI https://example.com | grep -i content-security-policy
```
