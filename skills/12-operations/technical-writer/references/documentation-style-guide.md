# Documentation Style Guide

> **Author:** Sandeep Kumar Penchala

A comprehensive style reference for technical writers — covering voice, structure, code, accessibility, tooling, and SEO. Companion to the [Technical Writer SKILL.md](../SKILL.md).

---

## 1. Voice and Tone

### Core Principles

| Principle | Do | Don't |
|---|---|---|
| **Professional but conversational** | "Let's configure the database connection." | "The database connection shall be configured by the user." |
| **Inclusive language** | "They can run the script" or restructure to avoid pronoun | "He runs the script" |
| **Second person ("you")** | "You'll need Node.js 18+ installed." | "The developer will need Node.js 18+ installed." |
| **Active voice** | "The CLI generates a config file." | "A config file is generated by the CLI." |
| **Present tense** | "The function returns a Promise." | "The function will return a Promise." |
| **No jargon without definition** | "Use an ORM (Object-Relational Mapper) like Prisma…" | "Leverage the ORM paradigm to abstract persistence…" |

### Avoiding Ableist and Exclusionary Language
- ❌ "crazy," "insane," "lame," "dummy value"
- ✅ "surprising," "unexpected," "placeholder value"
- ❌ "easy," "simple," "just" (assumes prior knowledge)
- ✅ State what's needed without judgment: "Install the package with:"

### Error Message Tone
- ❌ "Invalid input." / "Something went wrong."
- ✅ "The email address format isn't valid. Example: user@example.com"
- ✅ "Connection failed. Check that the server is running on port 3000 and your firewall allows outbound connections."

## 2. Document Types (Diátaxis Framework)

| Type | Goal | Reader Asks | Structure |
|---|---|---|---|
| **Tutorial** | Learn by doing | "How do I get started?" | Step-by-step; hands-on; minimal explanation |
| **How-To Guide** | Solve a problem | "How do I…?" | Goal-oriented; assumes existing knowledge; practical |
| **Reference** | Describe | "What does X do?" | Exhaustive; structured; lookup-friendly |
| **Explanation** | Understand | "Why does X work this way?" | Background, context, trade-offs; no step-by-step |
| **API Reference** | Look up endpoints | "What parameters does /users take?" | Auto-generated from spec; organized by resource |

### When to Use Each
```
"Beginner wants to learn"            → Tutorial
"User has a specific task"           → How-To Guide
"User needs exact parameter details" → Reference
"User wants to understand design"    → Explanation
"Developer integrating the API"      → API Reference
```

## 3. Structure Conventions

### Headings
```markdown
# Page Title (one per page — H1)
## Major Section (H2)
### Sub-section (H3)
#### Detail level (H4 — use sparingly)
```

- **Sentence case:** "Set up your development environment" not "Set Up Your Development Environment"
- **Action-oriented** for how-to guides: "Deploy to production"
- **Noun phrases** for reference: "Configuration options"
- Never skip heading levels (no H2 → H4 jump)

### Links
- ✅ Descriptive: "Read the [authentication guide](/docs/auth) for setup details."
- ❌ "Click here": "Click [here](/docs/auth) to learn about authentication."
- ✅ Link to the specific version: `/docs/v2/api/users` not `/docs/api/users`

### Lists
- Use **bulleted lists** for options, features, non-sequential items
- Use **numbered lists** for steps, ranked items, sequential procedures
- Keep list items parallel in structure (all start with verb, or all are noun phrases)

### Callouts (Admonitions)
```markdown
> **Note:** Useful supplementary information.
> **Warning:** Potential data loss or security risk.
> **Tip:** Best practice or time-saving suggestion.
> **Important:** Critical information that must not be missed.
```

## 4. Code Examples

### Standards
```markdown
1. Always include a language tag: ```python not ```
2. Code must be complete and runnable (include imports, error handling)
3. Use meaningful variable names: `user_list` not `ul`
4. Include error handling in production-ready examples
5. Comment the "why" not the "what"
```

### Code Block Template
````markdown
### Create a user
```python
import requests

def create_user(name: str, email: str, token: str) -> dict:
    """Create a new user account. Returns the user object on success."""
    response = requests.post(
        "https://api.example.com/v1/users",
        headers={"Authorization": f"Bearer {token}"},
        json={"name": name, "email": email},
        timeout=10  # Prevent hanging on network issues
    )
    response.raise_for_status()  # Raises HTTPError for 4xx/5xx
    return response.json()

# Usage
user = create_user("Jane", "jane@example.com", token="sk_abc123")
print(f"Created user: {user['id']}")
```
````

### Multi-Language Examples
When showing the same operation in multiple languages, use tabs:

````markdown
```python tab="Python"
# Python implementation
```

```javascript tab="Node.js"
// Node.js implementation
```

```go tab="Go"
// Go implementation
```
````

### Inline Code
- Use backticks for: commands (`npm install`), file paths (`config.yaml`), variable names (`api_key`), HTTP status codes (`200 OK`)
- Do NOT use backticks for: product names, people's names, ordinary nouns

## 5. Accessibility

### Alt Text
```markdown
✅ ![Sequence diagram showing OAuth 2.0 authorization code flow with client, auth server, and resource server](oauth-flow.png)
❌ ![screenshot.png](oauth-flow.png)
❌ ![Image](oauth-flow.png)
```

### Descriptive Link Text
```markdown
✅ See the [Kubernetes deployment guide](/docs/deploy/kubernetes) for container orchestration.
❌ For container orchestration, [click here](/docs/deploy/kubernetes).
❌ [https://docs.example.com/deploy/kubernetes](https://docs.example.com/deploy/kubernetes)
```

### Semantic HTML in Docs
```html
<!-- ✅ Correct heading hierarchy -->
<h1>Deployment Guide</h1>
  <h2>Prerequisites</h2>
  <h2>Deploy to AWS</h2>
    <h3>Configure IAM</h3>
    <h3>Set up ECS</h3>

<!-- ❌ Broken hierarchy -->
<h1>Deployment Guide</h1>
  <h3>Prerequisites</h3>  <!-- Skipped H2 -->
  <h4>Deploy to AWS</h4>  <!-- Skipped H2, H3 -->
```

### Color Contrast
- Code blocks: background must have ≥ 4.5:1 contrast with text
- Links: must be distinguishable from body text without relying on color alone (underline or bold)
- Diagrams: use patterns in addition to colors for color-blind readers

### Keyboard Navigation
- All interactive elements focusable and operable via keyboard
- Tab order follows visual order
- Skip-to-content link for screen readers

## 6. Information Architecture

### Progressive Disclosure
```
Level 1: Overview (1–2 sentences; what and why)
Level 2: Quickstart (5-minute path to first success)
Level 3: Detailed guides (by topic, by use case)
Level 4: Reference (complete API/config reference; look up, don't read)
```

### Navigation Patterns
- **Consistent:** Same categories in same order across all pages
- **Search-first:** Prominent search bar; predictive results; fuzzy matching
- **Breadcrumbs:** `Home > Guides > Authentication > OAuth 2.0 Setup`
- **Related links:** At bottom of each page: "Next: [Deploy to Production] / See also: [Security Best Practices]"

### URL Structure
```
✅ /docs/guides/authentication/oauth-setup
✅ /docs/api/v2/users/create
✅ /docs/reference/config-file
❌ /docs/page?id=42
❌ /docs/2024/march/new-auth-guide
```

## 7. Review Process

### Three-Tier Review

| Review Type | Who | Focus | Checklist |
|---|---|---|---|
| **Technical Review** | Engineer / SME | Accuracy; code works; steps complete | [ ] Code samples run without errors; [ ] All steps tested on clean install; [ ] API parameters match actual spec |
| **Editorial Review** | Writer / Editor | Clarity, style, structure | [ ] Follows style guide; [ ] No jargon without definition; [ ] Headings are descriptive; [ ] Links work |
| **Peer Review** | Another writer | Reader perspective | [ ] Can a new user follow this? [ ] Are assumptions explicit? [ ] Is the tone appropriate? |

### Review Checklist
```markdown
## Technical Review
- [ ] All code examples execute without errors on latest stable version
- [ ] Steps tested on macOS, Linux, and Windows (if applicable)
- [ ] API endpoints match current spec (version X.Y.Z)
- [ ] Error messages shown match actual error output
- [ ] Prerequisites list is complete

## Editorial Review
- [ ] One H1 per page; no skipped heading levels
- [ ] Active voice; second person; present tense
- [ ] All acronyms spelled out on first use
- [ ] Links use descriptive text (no "click here")
- [ ] Admonitions used correctly (Note/Warning/Tip/Important)
- [ ] Inclusive language throughout

## Accessibility Review
- [ ] All images have descriptive alt text
- [ ] Color contrast passes WCAG AA (≥ 4.5:1 for text)
- [ ] Semantic heading hierarchy maintained
- [ ] Keyboard-navigable interactive elements
```

## 8. Tooling

### Docs-as-Code Stack

```yaml
# Core tools
authoring: Markdown + MDX
version_control: Git (GitHub/GitLab)
static_site: Docusaurus / Nextra / Mintlify / Starlight

# Quality
linting: Vale (prose lint) + markdownlint
spellcheck: cSpell or LanguageTool
link_checker: lychee or markdown-link-check
preview: Local dev server (hot reload)

# Diagrams
diagrams: Mermaid (text-based) + Excalidraw (hand-drawn style)
screenshots: CleanShot X or Shottr (macOS)

# Publishing
ci_cd: GitHub Actions → deploy to Vercel / Cloudflare Pages / Netlify
analytics: Plausible / PostHog (privacy-friendly) or Google Analytics
search: Algolia DocSearch or Pagefind
```

### Static Site Generator Comparison

| Tool | Best For | Key Strength | Pricing |
|---|---|---|---|
| **Docusaurus** | Open source; React ecosystem | Versioning, i18n, plugin ecosystem | Free (OSS) |
| **Nextra** | Next.js projects; clean design | MDX, fast, file-system routing | Free (OSS) |
| **Mintlify** | API-first products | Beautiful API ref; OpenAI-powered search | Free + paid tiers |
| **Starlight** | Astro projects; accessible by default | Built-in accessibility, fast | Free (OSS) |
| **GitBook** | Non-technical teams | WYSIWYG, collaboration, change requests | Free + paid tiers |

## 9. SEO for Documentation

### Page-Level SEO Checklist

| Element | Best Practice | Example |
|---|---|---|
| **Title tag** | `[Topic] — [Product] Docs` | `Authentication — Acme API Docs` |
| **Meta description** | 150–160 chars; includes primary keyword; compels click | "Learn how to authenticate users with OAuth 2.0, API keys, and JWTs. Step-by-step guide with code examples in Python and Node.js." |
| **H1** | One per page; matches title tag intent | `# OAuth 2.0 Authentication` |
| **URL** | Slug matches H1 (kebab-case) | `/docs/auth/oauth-2-0` |
| **Internal links** | 3–5 contextual links to related pages | Linked from: Quickstart, Security, API Reference |
| **Canonical URL** | Set if content exists at multiple URLs | `<link rel="canonical" href="https://docs.example.com/auth/oauth" />` |

### Sitemaps
```xml
<!-- Submit to Google Search Console + Bing Webmaster Tools -->
<!-- Static site generators generate sitemap.xml automatically -->
<urlset xmlns="http://www.sitemaps.org/schemas/sitemap/0.9">
  <url>
    <loc>https://docs.example.com/guides/auth</loc>
    <lastmod>2024-03-15</lastmod>
    <changefreq>monthly</changefreq>
    <priority>0.8</priority>
  </url>
</urlset>
```

### Robots.txt
```
User-agent: *
Allow: /
Sitemap: https://docs.example.com/sitemap.xml
Disallow: /api/  # Block auto-generated API reference if duplicate content concern
```

### Measuring Doc SEO
- **Organic traffic** to docs subdomain (Google Search Console)
- **Top queries** driving doc traffic
- **Click-through rate** from SERP to docs
- **Bounce rate** on doc pages (high bounce on reference pages is normal; high bounce on tutorials = problem)
